# PRD: 星星云宠物网页版 MVP

状态：Draft  
日期：2026-06-21  
Owner：Product Team  

## Overview

### Problem Statement

家长想帮助孩子建立学习习惯，但日常提醒容易演变成催促和冲突。孩子需要的是即时、温和、可爱的反馈；家长需要的是低维护、隐私可控、不会制造焦虑的工具。

### Solution Summary

构建一个手机优先的网页产品：孩子完成阅读、作业、整理、复盘等任务后获得成长能量，专属宠物“星星”成长、改变心情、解锁房间物品并生成温和日记。家长通过家长中心设置任务、确认关键完成、查看趋势、导出/删除数据。

### Target Users

- Primary：小学阶段孩子，使用孩子端完成任务和陪宠物成长。
- Secondary：家长，配置任务、确认、管理隐私和试用效果。

## Goals & Success Metrics

### Goals

1. 让孩子连续 7 天愿意打开并完成至少 1 个学习习惯任务。
2. 让家长每天维护成本低于 2 分钟。
3. 建立安全、无压力、可审计的儿童友好反馈机制。

### Success Metrics

| Metric | Baseline | Target | Timeline |
| --- | --- | --- | --- |
| 7 日活跃天数 | Unknown | >= 5/7 天 | 首轮家庭试用 |
| 首次任务完成率 | Unknown | >= 80% | 首轮家庭试用 |
| 家长每日维护时间 | Unknown | P75 <= 2 分钟 | 首轮家庭试用 |
| 孩子负面压力反馈 | Unknown | 0 明显焦虑案例 | 持续 |
| PWA/网页首屏加载 | Unknown | 4G 下 <= 3 秒 | MVP QA |

### Non-Goals

- 不做微信小程序。
- 不做班级、老师端、多学生管理。
- 不做排行榜、公开社交、商城、抽奖。
- 不做自由 AI 聊天。
- 不做学科内容库或自动批改。

## Scope

### In Scope

- Responsive web app for mobile and desktop.
- Child-facing pet room with today tasks.
- Parent-facing task management and confirmation.
- Deterministic pet state engine.
- Rule-based safe message templates for encouragement, summary, story, reminder.
- Local-first storage by default.
- Export/delete data controls.
- Stable deployment preview URL.
- Product analytics spec without collecting unnecessary child data.

### Out of Scope

- 微信小程序 AppID、云函数、微信审核。
- Payment, subscription, ads.
- Public community or sharing.
- AI free chat or voice input.
- School, teacher, class roster.
- Camera, microphone, location, contacts.

### Future Considerations

- PWA install prompt and offline mode.
- Optional parent-auth cloud sync.
- Custom pet asset packs generated by `hatch-pet`.
- Carefully gated AI story generation after safety testing.
- Multi-child family profiles.

## Solution Design

### Functional Requirements

#### Onboarding

- FR-001: The product must let a parent start with default profile, default pet, and default tasks in under 2 minutes.
- FR-002: The parent may edit child nickname, pet name, grade band, and default task set.
- FR-003: The product must show a child-friendly first-run pet greeting after setup.

#### Daily Task Board

- FR-010: The child must see 1-5 active tasks for today, with default recommendation capped at 3.
- FR-011: Each task must include title, category, estimated duration, energy value, and status.
- FR-012: Completing a no-approval task immediately grants energy up to the daily cap.
- FR-013: Completing an approval-required task puts it into pending parent confirmation.
- FR-014: A task can be undone by the parent or child before daily close.

#### Pet State Engine

- FR-020: Pet stage is determined by cumulative growth energy.
- FR-021: Daily mood is determined by completed tasks, pending approvals, and today energy.
- FR-022: No pet state may imply death, illness, punishment, abandonment, or blame.
- FR-023: The engine must expose deterministic snapshots for testing.

#### Parent Center

- FR-030: Parent can create, edit, disable, and reorder task templates.
- FR-031: Parent can approve pending tasks in one list.
- FR-032: Parent can view 7-day and weekly summaries.
- FR-033: Parent can export local data as JSON.
- FR-034: Parent can delete local data and restart.

#### Pet Diary

- FR-040: Product must generate rule-based encouragement, daily summary, weekly summary, and short story.
- FR-041: Diary entries must be stored locally and visible to parent.
- FR-042: AI-generated text, if added later, must pass safety filters and be marked as AI-generated.

#### Privacy and Safety

- FR-050: Product must not request camera, microphone, contacts, location, or school identity.
- FR-051: Product must provide a parent-readable privacy page before any cloud sync.
- FR-052: Product must not collect more child data than needed for the habit loop.
- FR-053: Product must offer export/delete controls before any public release.

### User Experience

The first screen should be the product experience, not a marketing landing page. On mobile, the child sees the pet room first, then today tasks. Parent controls are accessible but visually secondary and may require a parent gate in a later version.

### Edge Cases

| Scenario | Expected Behavior |
| --- | --- |
| No tasks today | Show a calm empty state and allow parent to add a task |
| Daily energy cap reached | Tasks can still complete but show no extra energy today |
| Parent approval pending | Pet acknowledges effort but waits for confirmation |
| Offline/local storage unavailable | Show recovery message and do not pretend save succeeded |
| Child repeatedly taps complete | Debounce and grant energy once |

## Technical Considerations

### Constraints

- Web-first. Remove or isolate mini program code before production implementation.
- Local-first MVP. No backend required for first family trial.
- All core rules must be unit-testable outside the UI.
- UI must be mobile-first and accessible.
- Deployment must provide stable HTTPS preview, not localtunnel-style temporary links.

### Integration Points

- Browser local storage or IndexedDB for MVP.
- Optional future auth/cloud storage.
- Optional future analytics with privacy-preserving event names and no child PII.

### Data Requirements

Data entities: ChildProfile, PetProfile, TaskTemplate, DailyTask, CheckIn, EnergyLedger, PetStateSnapshot, ParentSettings, DiaryEntry, SafetyAuditLog.
